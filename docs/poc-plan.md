# PoC 게이트 1: 한글 글리프 가시성 + 대사 2바이트 경로 판정

> **결과 (1차 부팅)**: 프로브 A/B → 대사창에 `@A`로 표시. 판정: 영문 패치 대사 엔진은 1바이트 VWF이며 Krom2RawAdd를 호출하지 않음(훅 미발화). 0x8840/0x8841이 '@','A'로 나온 것은 상위 바이트 무시+하위 바이트 ASCII 렌더 또는 ≥0x80 스킵. 아래 v2로 전환.

## PoC v2 (현행): 아틀라스 셀 교체 + 뱅크 인코딩 (코드 무수정, 순수 데이터)

확정된 엔진 구조 (EXE 디스어셈블):
- 대사 렌더러(0x80068xxx 루프)는 1바이트 글리프 코드 + `00 4X` 뱅크 제어코드로 16비트 코드(뱅크<<8|바이트)를 만들고, 드로우 함수 0x80066D38이 **256×256 4bpp 아틀라스(10×10 셀, 행당 25개)**에서 SPRT로 찍는다. UV가 u8이라 1페이지=625셀 한계. 코드 0x2FF는 특수(아이콘), 0x1CA/0x1D0/0x219는 y오프셋 특례.
- 아틀라스 = `I1/WINTIM*.AR`(창 테마 5종) 내 `1A_BMP.TIM` (VRAM 640,256). 텍스페이지 gp+0x2648(gp=0x800BA3CC), 로더 0x80066520이 이름("FNT0.TIM","FNT_DCOL.TIM","1A_BMP.TIM")으로 조회·업로드. FNT0.TIM은 영문 패치에서 8×8 스텁(VRAM 768,256) — 원판은 대형 폰트 시트였을 것(2페이지 확장 후보 자리).
- 폭: EXE 0xA3A00(RAM 0x800B3200) 256엔트리 테이블[바이트], ≥0x80은 10. 폭 셈 0x800B508C가 gp+0x2070에 기록.
- 원본 일본어판도 같은 구조(아틀라스의 한자 = 게임 전체 한자 서브셋, KANJI.MES 등은 별도 역할).

v2 프로브 (모두 데이터 패치, EDC/ECC 재계산 완료):
- 아틀라스 셀 0x180/0x181(한자 셀)과 0x141/0x142에 '한'/'글' 10×10 도트 글리프 기입 (5개 테마 전부).
- GEVMSG08.MES 오프닝 대사: `[C] 00 41 80 81 00 40` (뱅크1+바이트 0x80/0x81) + `[D] 00 41 41 42 00 40` (뱅크1+바이트 0x41/0x42).

## ✅ PoC 게이트 통과 (v4, 2026-07)

세이브스테이트 VRAM 실측으로 대사 폰트 소스를 확정하고, 임베드 폰트의 'A'/'B' 글리프를 한/글로 교체 → **대사창에 한글 정상 표시 확인**.

확정 사실:
- **대사 폰트 = 각 대사 MES 파일에 임베드된 256×96 4bpp ASCII VWF 시트(12,288 B, 128 B/행)**. 로드 시 VRAM(768,256)=page0에 업로드. 셀 격자 16×16px, 16열, ASCII 0x20~0x7F(첫 글자행 0x20 기준 col=(c-0x20)%16, row=(c-0x20)//16). MES 내 오프셋은 파일마다 다름(GEVMSG08=0x2BD8, MB0=0x1F30, DMESPL=0x2834 등). 비압축 직접 업로드 — raw 패치로 교체 가능.
- **글리프 본체 색 = CLUT index 1**(진한색). 가장자리에 상위 index로 AA. 한글 글리프도 index 1로 그리면 원본과 동일한 검정.
- **page1(VRAM 832,256)은 완전히 비어 있음** → 한글 전용 시트를 얹을 자리. code≥0x100(뱅크 1+)이면 렌더러가 page1 텍스처페이지(gp+0x2638[1]=TPage 0x1D)를 자동 선택하므로, page1에 한글 시트를 올리고 텍스트에 `00 41`(뱅크1) + 바이트를 쓰면 됨.
- 세이브스테이트 VRAM 추출법 확립: gzip 해제 → EXE 코드로 RAM 위치 → 1A_BMP 아틀라스 시그니처의 VRAM stride(2048B) 검증으로 vram_base 특정(`tools/sstate_find_vram.py`, `sstate_vram.py`). 이 스테이트 vram_base=0x2733DF.

## 본 패치 아키텍처 (확정)
1. **한글 시트를 page1(832,256)에 추가** — 각 대사 MES의 폰트 임베드 영역을 page0(ASCII 유지) + page1(한글) 2페이지로 확장. code = 0x100|바이트로 한글 접근.
2. 렌더러는 이미 page 비트(code&0x300) 처리 → 코드 수정 최소. U좌표 테이블(0x800BF920)과 폭 테이블(0x800B3200)에 한글 코드 엔트리 추가 필요 여부 확인.
3. 텍스트: 한글 음절 → (뱅크1 프리픽스 + 바이트) 인코딩. 1페이지 256셀 → 한글 음절 부분집합. 부족하면 page2(896,256, TPage 0x1E)까지 3페이지 = ~750셀.
4. **선결 조사**: (a) 227개 대사 MES 전부가 폰트를 임베드하는지/압축인지, (b) MES 내 폰트 블록 위치 규칙, (c) 폰트 임베드가 파일별 중복이면 전 MES 일괄 주입 도구 필요.

## 폰트 업로드 함수 (0x8006AAE0, 디스어셈블 확정)

각 MES 로드 시 임베드 폰트 블록을 LoadImage로 VRAM에 올린다:
1. LoadImage rect{x=768, y=256, w=64(=256px 4bpp), h=256}, data=블록+0xC → **page0 (256×256)**. ASCII는 위 96행만, **아래 160행 여유**.
2. data+=0x8000; rect{x=832, y=256, w=64, h=128}, → **page1 (256×128)**. **현재 데이터가 0(빈) = 한글 자리, 코드 수정 불필요**.
3. data+=0x4000; `alloc(0x180)` 후 그 포인터를 gp+0x2664에 저장, 블록에서 0x180(384)B 복사 = **glyph metric 테이블**(code별 폭/UV, 렌더러 0x80067F70이 참조). 추가로 s3(page0 원본 포인터)에서도 0x180 복사.

임베드 폰트 블록 전체 구조(파일 내): `[0xC 헤더] [page0 0x8000] [page1 0x4000] [metric 0x180 ...]`. 226 MES 전부 동일 블록.

**한글 주입 전략(코드 무수정)**: 각 MES 임베드 블록의 page1 영역(오프셋 +0xC+0x8000, 256×128 4bpp)에 한글 글리프 시트를 기입 + metric 테이블에 한글 code 폭 설정. 텍스트는 `00 41`(뱅크1) + 바이트로 한글 접근(code=0x100|byte). page1 128행/16px = 8행×16열 = 128셀 + page0 여유 160행(10행×16열=160셀) = ~288 한글 슬롯(15px 기준). 부족 시 글리프 축소(12~13px)로 행 증대 또는 자주 쓰는 음절 부분집합.

### ✅ page1 뱅크1 경로 확정 (PoC v5, 2026-07)

임베드 폰트 블록의 **빈 page1에 한글을 넣고 뱅크1로 호출 → 대사창에 '한글' 렌더 성공**. 엔진/EXE 무수정.

확정된 한글 인코딩(모두 MES 파일 내에서 완결):
- **폰트 블록 정밀 구조** (파일 내, 블록 시작 = ASCII 글리프 위치 − 0x1000): `page0 @블록+0x0 (0x8000, VRAM 768,256)` + `page1 @블록+0x8000 (0x4000, VRAM 832,256, 비어있음)` + `metric @블록+0xC000 (0x180)`. 블록 끝 = 파일 끝. (GEVMSG08 예: 블록 0x1BD8, page0글리프 0x2BD8, page1글리프 0xABD8, metric 0xDBD8.)
- **셀 격자**: 16×16px, 16열, 8바이트/행(4bpp). page0/page1 공통. code 하위바이트 B에 대해 row=(B−0x20)//16, col=(B−0x20)%16. glyph 본체 CLUT index 1.
- **인코딩**: 텍스트 `00 41`(뱅크1 전환, 상위니블 0x4 = 뱅크셋, 하위니블 = 뱅크값) → 이후 바이트 B는 code=0x100|B로 page1 렌더. 뱅크0 복귀는 `00 40`.
- **metric 테이블**(블록+0xC000, 0x180=384B): metric[code] = 폭(px). ASCII는 0x20~0x7E에 값(A=13,i=3,space=4 등), 한글 code(0x141 등)에 폭 기입 필요. 렌더러 폭 참조.
- U좌표: 렌더러가 page 비트(code&0x300)로 텍스처페이지 선택(page1=832,256), within-page는 격자. (0x800BF920 EXE 테이블은 page1 byte 0x20~0x7F 재사용 시 그대로 유효 — EXE 수정 불필요.)

**남은 것**: 226 MES 일괄 한글 주입 도구(page1 시트 + metric) + 한글 code 할당(byte 0x20~0x7F = 96셀/page1, page0 여유행·page2 확장 시 증대). 슬롯 부족분은 자주 쓰는 음절 우선.

### 재삽입 설계 확정 (파일 크기 유지 in-place)

- **폰트 블록 = 마지막 텍스트 레코드 직후**에 배치됨(MB0: 레코드 끝 0xF30 = 블록 시작 0xF30). 런타임은 레코드를 순회하다 size 필드가 0(=폰트 블록 page0 row0의 빈 데이터)에서 멈추므로, 폰트 위치는 텍스트 길이에 따라 자동 결정 — 헤더에 폰트 오프셋 필드 없음(폰트 로드 함수 0x8006AAE0은 간접 호출).
- **MES 파일은 뒤에 대량 제로 패딩**(예: MB0 53,424B 중 텍스트+폰트 0xCF30 뒤 나머지 제로). 재인코딩 텍스트 + 폰트블록(0xC180 고정) ≤ 원본 파일 크기이면, **폰트 블록 뒤 제로를 조절해 파일 크기를 원본과 동일하게 유지** 가능.
- **page1 한글·metric 채우기는 블록 크기 불변**(빈칸 채움). 텍스트만 재인코딩.
- 결과: 파일 크기 원본 동일 → **ISO 재배치·EMBLEM.FAT·디렉토리 갱신 불필요, in-place 섹터 교체 + EDC/ECC만**(tools/img_patch.py 그대로). 한글이 영문보다 대체로 짧아 크기 초과 위험 낮음(초과 시 그 파일만 디스크 끝 재배치).
- **MES별 로컬 폰트**: 각 MES가 독립 폰트 블록을 가지므로, 파일별로 그 대사의 고유 음절만 page1에 담으면 됨(전역 1000자 불필요, 파일당 수십~백여 음절 → 96셀+page0여유로 커버). 슬롯 초과 파일만 page0 여유행/축소 폰트 적용.

### 주입 도구 파이프라인 (mes_inject, 예정)
입력: MES + 레코드별 한글 번역 → (1)고유 음절 수집·byte code 할당 (2)page1 글리프 렌더(16×16, index1) + metric 폭 기입 (3)텍스트 재인코딩(제어코드 보존, 텍스트 세그먼트를 뱅크1+byte로, 뱅크 전환 관리) (4)레코드 재조립 + 폰트블록 + 제로패딩(원본 크기) → img in-place 교체.

## 훅 구성 (EXE in-place, 동일 크기)

- B(0x51) Krom2RawAdd 스텁(RAM 0x800A5860, EXE 오프셋 0x96060) → `j 0x800CCE00`.
- 케이브 0x800CCE00 (EXE 꼬리 제로 패딩 0x800CC947~0x800CD000 내): lead 0x88~0x8E && lo≥0x40이면 `v0 = 0x800CCE80 + ((lead-0x88)*188 + lo_idx) * 30` 반환, 아니면 원래 BIOS 트램폴린 폴스루.
- 폰트 0x800CCE80: '한'(0x8840), '글'(0x8841) — 15×15 1bpp, 행당 u16 BE, bit15=왼쪽(렌더러 0x80016F2C 소비 방식 실측으로 포맷 확정).
- EDC/ECC: 수정 섹터(1046, 1124, 232064) 재계산·검증 완료. 도구 `tools/cd_edcecc.py`는 원본 무수정 섹터 50개 재계산 대조로 자가 검증됨.
