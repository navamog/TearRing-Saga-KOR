# SLPS-03177 (티어링사가 유토나 영웅전기, 영문패치판) 초기 조사

## 보유 매체

| 항목 | 값 |
| --- | --- |
| 이미지 | `티어링사가(eng).img` |
| 크기 | 594,054,048 B = 252,574 섹터 × 2352 B (raw Mode2/2352, 단일 데이터 트랙) |
| 볼륨 ID | `TEARRINGSAGA`, PVD LBA 16, 루트 디렉토리 LBA 22 |
| 부트 | `SYSTEM.CNF` → `cdrom:\SLPS_031.77;1` (SLPS-03177) |
| 상태 | 영문 팬패치 적용판. 수정 파일은 디스크 끝(LBA ~230000+)에 재배치되고 디렉토리 레코드만 갱신됨. 원위치에 구버전 사본이 잔존(고아 섹터) |

원본 일본어판 이미지는 보유하지 않음. 한글 패치의 베이스는 이 영문패치판.

## 실행 파일

- `SLPS_031.77`: PS-X EXE, load 0x80010000, text_size 0xBD000, PC 0x8009ADD0, SP 0x801FFFF0.
- RAM 주소 ↔ 파일 오프셋: `file = ram - 0x80010000 + 0x800`.
- EXE 내부에 영문 시스템 텍스트 존재(LBA 1085 부근 = 파일 0xA9CC8 부근, 이름 테이블 추정) — 영문 패치는 EXE를 동일 크기 그대로 수정.

## 파일시스템 (확인 사실)

- ISO9660 표준 워크로  7,526개 파일 파싱 완료. 총 22개 디렉토리.
- `EMBLEM.FAT` (204,800 B) = 게임 자체 파일 테이블. **25 B 엔트리 × 8192**: flag(1) + name(16, null-pad) + LBA(u32 LE) + size(u32 LE). B.BIN LBA/크기와 일치 확인. 파일 재배치 시 이 테이블 갱신 필요(영문 패치도 이 방식일 것 — 검증 필요: 재배치 파일의 FAT 엔트리가 새 LBA인지 확인할 것).
- 오버레이: `B.BIN`("MI.BD" 매직, 디버그 문자열 포함), `E/G/H/J/T.BIN`. `H.BIN`은 RAM 0x800F2xxx대 점프테이블로 시작하는 코드/데이터(영문 패치가 수정·재배치, LBA 236286).
- `TMP.CTI`: CD 마스터링 레이아웃 스크립트 잔재(수정 불요).

## 텍스트 위치 (확인 사실)

- 본문 스크립트: `MB1/*.MES`(맵/전투 대사, 156개), `MG1/GEVMSG*.MES`, `EM1/`, 기타 MES 계열. 매직 `ESMD`.
- 영문 패치 텍스트는 1바이트 ASCII + `00 xx` 제어코드(예: `00 10` 개행 추정, `00 70`/`00 80` 박스 제어, `00 50 02 xx` 화자/초상 추정, `00 5D 01 xx` 미상). 원문(일본어판)은 2바이트 SJIS였을 것으로 추정(가설).
- `MB1/DMESPL.MES` 죽음 대사 확인. `MB1/KANJI.MES`는 별도 타입(헤더 필드 0x1C), 역할 미확정(가설: SJIS↔글리프 매핑 또는 한자 목록).
- EXE 내 이름/시스템 텍스트 테이블 존재.

## 폰트 경로 (확인 사실)

- EXE에 BIOS B-table 스텁: **0x800A5860 = B(0x51) Krom2RawAdd**, **0x800A5870 = B(0x5B) Krom2RawAdd2**. 게임은 BIOS 내장 SJIS 폰트(15×15 1bpp)를 사용.
- 호출 사이트: 0x800A5860 ← EXE 0x76F0(=RAM 0x80016EF0), 0x7920. 0x800A5870 ← 0x8B91C, 0x8BE58, 0x99324, 0x993B8, 0x99740, 0x997D0, 0x9982C (파일 오프셋).
- 렌더러 0x80016EA0: `lbu` 2회로 [lead, trail] BE 조립 → Krom2RawAdd → 스크래치패드 0x1F80007C/0x1F800004 경유 1bpp→변환(서브루틴 0x8001735C) . 진입부에서 lead < 0x80이면 별도 경로 분기(영문 패치 1바이트 경로 추정 — 원본에도 있었는지 미확정).

## 라운드트립 상태

- 추출: raw 섹터 user data(offset 24, len 2048) 절취로 ISO9660 파일 추출 확인.
- 재삽입: 동일 크기 in-place 교체 + 해당 섹터 EDC/ECC 재계산 경로로 설계 예정(미검증 → PoC 게이트).
- 크기 변경 시: 영문 패치와 같은 "디스크 끝 재배치 + 디렉토리 레코드/EMBLEM.FAT 갱신" 경로 사용 가능(영문 패치가 이미 실증).

## 한글화 전략 초안 (가설, PoC 대상)

1. 선행 사례(PS1 문서 6절)와 동일하게 SJIS 한자 영역(0x88~0x8E, 1,316슬롯+확장 여지)을 한글 음절에 재할당.
2. Krom2RawAdd/Krom2RawAdd2 스텁(0x800A5860/70) 또는 호출 사이트에 훅 → lead 0x88+면 자체 한글 폰트(15×15 1bpp, 30 B/자) 포인터 반환.
3. 폰트 적재: EXE 확장 또는 미사용 RAM 영역 + 디스크 추가 섹터(영문 패치 방식대로 끝에 배치).
4. 텍스트: 영문 MES를 파싱해 한글 재삽입(2바이트 경로). 영문 패치가 추가한 1바이트 경로와의 공존 확인 필요.

## 미해결 (다음 게이트)

1. MES 컨테이너 구조(엔트리 테이블·오프셋 방식) 확정 → 라운드트립 파서.
2. 렌더 경로 전체 콜체인: 대사창이 어느 함수·어느 스텁을 쓰는지, ASCII 경로의 글리프 출처(영문 패치 자체 폰트?) 확정.
3. 동일 크기 in-place 재삽입 + EDC/ECC 재계산 검증(에뮬레이터 부팅).
4. 한글 글리프 예산: 번역 코퍼스 기반 고유 음절 수 산정(번역 착수 후 반복).
